// 01 — Système de Couleurs

Les règles de couleur qui font la différence

Un vrai système professionnel ne s'improvise pas. Il repose sur 3 niveaux de tokens : Primitifs → Sémantiques → Composants. C'est ce que font Vercel, Linear, IBM Carbon, Google Material.

🎨

La règle des 3 niveaux — la clé de tout

Primitifs : les valeurs brutes (#d4a853, #16a34a). Sémantiques : la signification (color.success, color.brand). Composants : l'usage précis (button.primary.bg). Changer un primitif = tout le système se met à jour. C'est comme ça que Vercel et Linear font leur dark mode en une ligne.

Architecture feature gating — Sans if(plan==='pro') partout

Le problème avec la plupart des implémentations : les vérifications de plan parsèment tout le code comme des mines. La solution : un hook centralisé.

// lib/plans/features.ts — Source de vérité unique // JAMAIS if(user.plan === 'pro') dans les composants export const FEATURES = { contacts_limit: { free: 50, pro: 500, expert: Infinity, clinique: Infinity }, ai_coach_monthly: { free: 0, pro: 20, expert: Infinity, clinique: Infinity }, academy_max_level: { free: 1, pro: 3, expert: 5, clinique: 5 }, pipelines_limit: { free: 1, pro: 3, expert: Infinity, clinique: Infinity }, email_sequences: { free: false, pro: 5, expert: Infinity, clinique: Infinity }, api_access: { free: false, pro: false, expert: true, clinique: true }, } as const export type Plan = 'free' | 'pro' | 'expert' | 'clinique' export type Feature = keyof typeof FEATURES // Hook React — LE seul endroit où vérifier le plan export function useFeature(feature: Feature) { const { plan } = useAuth() // depuis Supabase const limit = FEATURES[feature][plan] return { limit, isAllowed: limit !== false && limit !== 0, isUnlimited: limit === Infinity, canDo: (current: number) => limit === Infinity || current < limit } }

Composant <FeatureGate> — Wrapping universel avec UX de verrouillage.

🔐 Auth robuste — Tokens courts + Rotation

04
RLS

🛡️ Supabase RLS — Isolation totale par utilisateur

Row Level Security sur CHAQUE table. Un utilisateur authentifié ne peut physiquement pas voir les données d'un autre — même en manipulant les requêtes API. La DB renvoie 0 résultats si l'auth.uid() ne correspond pas. C'est la principale défense contre les attaques IDOR (Insecure Direct Object References) — le risque #1 OWASP. Audit mensuel : supabase inspect db --security pour vérifier qu'aucune table n'a oublié RLS.

Anti-Scraping

🤖 Détection d'exfiltration de data

Anomaly detection sur les patterns de requêtes : si un compte fait plus de 50 requêtes GET sur /api/contacts/ en 5 minutes = exfiltration probable → alerte + throttling agressif. Pagination obligatoire (max 100 résultats par requête, jamais de SELECT sans LIMIT). Les endpoints exportCSV sont limités à 1 export/heure par compte et loggés. Response body size monitoring : si un endpoint retourne plus de 500ko = alert.

// middleware.ts — Protection centralisée sur toutes les routes /api/* import { Ratelimit } from '@upstash/ratelimit' import { Redis } from '@upstash/redis' const ratelimit = { free: new Ratelimit({ redis: Redis.fromEnv(), limiter: Ratelimit.slidingWindow(10, '1m') }), pro: new Ratelimit({ redis: Redis.fromEnv(), limiter: Ratelimit.slidingWindow(60, '1m') }), expert: new Ratelimit({ redis: Redis.fromEnv(), limiter: Ratelimit.slidingWindow(200, '1m') }), clinique:new Ratelimit({ redis: Redis.fromEnv(), limiter: Ratelimit.slidingWindow(500, '1m') }), } export async function middleware(req: NextRequest) { // 1. Routes publiques → skip if (!req.nextUrl.pathname.startsWith('/api/')) return // 2. Identifier l'utilisateur et son plan depuis le JWT const session = await getSessionFromRequest(req) if (!session) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 }) // 3. Rate limiting selon le plan const plan = session.user.plan ?? 'free' const { success, limit, reset, remaining } = await ratelimit[plan].limit(session.user.id) if (!success) { // 4. Log tentative d'abus pour audit await logAbuseAttempt(session.user.id, req.nextUrl.pathname) return NextResponse.json( { error: 'Rate limit exceeded', plan_limit: `${plan}: ${limit} req/min` }, { status: 429, headers: { 'Retry-After': String(Math.ceil((reset - Date.now()) / 1000)), 'X-RateLimit-Limit': String(limit), 'X-RateLimit-Remaining': String(remaining), } } ) } // 5. Ajouter le plan dans les headers pour les API Routes const response = NextResponse.next() response.headers.set('X-User-Plan', plan) return response }
