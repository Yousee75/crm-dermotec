// 02 — Priorités par impact

Ce que tu fais en premier

Classé par ratio impact/effort. Les 4 premières actions donnent 80% des gains de performance.

1

React Server Components — déjà en place

Les Server Components ne s'envoient PAS au client. Bundle JS réduit de 30-62%. LCP amélioré de 65%. INP de 250ms → 175ms. C'est le gain le plus massif avec le moins d'effort sur Next.js 15.

−62% bundle−65% LCP
2

Images AVIF + next/image priority

AVIF = −50% vs JPEG à qualité identique. next/image génère automatiquement WebP/AVIF, ajoute width/height (anti-CLS), lazy-load automatique. Pour l'image LCP hero : ajouter priority={true}.

−50% poidsCLS=0
3

Partial Prerendering (PPR) — Next.js 15

Le shell statique de la page (sidebar, header, layout) est servi depuis le CDN en <50ms. Les données dynamiques (dashboard, points) streamées derrière. Le pro voit immédiatement l'interface, pas un écran blanc.

ExpérimentalShell <50ms
4

Fonts — next/font + display swap

next/font auto-héberge Cal Sans et Geist, génère les fallback metrics automatiquement, évite le CLS de font. Zero layout shift garanti. La font est préchargée avant le premier paint.

CLS=0Preload auto
5

useTransition + useDeferredValue

Pour les actions CRM coûteuses (filtrer 247 contacts, chercher, ouvrir une modal lourde) → marquer comme "non-urgent". React traite d'abord l'UI, le reste après. INP garanti <100ms même sur desktop bas de gamme.

INP <100msReact 19
6

Virtualisation listes longues

La table contacts (247+ entrées) et le feed activités : utiliser TanStack Virtual (react-virtual). DOM = seulement les lignes visibles. Interaction instantanée même avec 10 000 contacts.

DOM minimal@tanstack/virtual
7

Edge caching Vercel + Cloudflare

Vercel Edge Network sert depuis le POP le plus proche. Pour le CRM : dashboard statique depuis edge, données dynamiques (Supabase) depuis origin. TTFB de 300ms → 20ms pour le shell.

TTFB <20ms300+ PoPs
8

Supabase queries — index + select partiel

Chaque query non-optimisée retarde le streaming. Index sur user_id, video_id, created_at. Jamais de SELECT * — sélectionner uniquement les colonnes nécessaires. Connection pooling avec pgBouncer (inclus Supabase Pro).

DB <10mspgBouncer

// 05 — Supabase Performance

Base de données ultra-rapide

🔗

Connection Pooling — 30min

Utiliser le port 6543 (pgBouncer transaction mode) au lieu de 5432. Élimine 50-200ms de cold start par requête serverless. Critique pour les API Routes Next.js.

DATABASE_URL:6543−200ms
📊

Index stratégiques — 2h

Index sur les colonnes les plus filtrées : user_id, created_at DESC, is_published. Analyser les slow queries avec supabase inspect db --verbose.

-- Index performance critiques CREATE INDEX idx_watch_user ON watch_events(user_id, created_at DESC); CREATE INDEX idx_videos_level ON videos(required_level, is_published); CREATE INDEX idx_points_user ON points_history(user_id) INCLUDE (points_earned);
🎯

Select partiel — toujours

Jamais SELECT * sur une table lourde. Sélectionner uniquement les colonnes nécessaires pour l'affichage. Les colonnes binaires (vdocipher_id) coûtent cher si non utilisées.

select('id, title, niveau')−40% payload
⚡

Parallel fetching

Ne jamais enchaîner des requêtes Supabase séquentiellement. Utiliser Promise.all pour les requêtes indépendantes. KPIs + modules + activité = 1 round-trip au lieu de 3.

// ❌ Séquentiel : 150ms + 120ms + 80ms const kpis = await getKpis() const modules = await getModules() // ✅ Parallèle : max(150, 120, 80) = 150ms const [kpis, modules] = await Promise.all([ getKpis(), getModules() ])
🔄

Realtime sélectif

Supabase Realtime pour les points uniquement (gamification). Pas de subscription inutile sur les tables statiques. Un channel Realtime = une connexion WebSocket permanente = overhead.

1 channel maxDéconnecter à unmount
🗄️

Materialized Views

Le tableau de bord KPI (48 200€ revenus, 247 contacts actifs) = aggréger plusieurs tables. Créer une Materialized View rafraîchie toutes les 5 minutes plutôt que calculer à chaque requête.

5min refresh−90% query time
// 06 — Outils de mesure

Comment mesurer et monitorer

  • 🔍
    PageSpeed Insights (PSI) pagespeed.web.dev — mesure LCP, INP, CLS sur les vrais utilisateurs Chrome. Utiliser avant ET après chaque optimisation. C'est la source officielle Google.
  • ⚡
    Vercel Speed Insights Intégré directement dans le dashboard Vercel. Mesure les Core Web Vitals sur vos vrais utilisateurs en continu. Alertes si une métrique dégrade.
  • 📦
    Bundle Analyzer — bundlejs.com Analyser le poids de chaque import AVANT de l'ajouter. Alternative : @next/bundle-analyzer. Objectif : <150ko JS par route (gzippé).
  • 🎯
    web-vitals library Mesurer LCP, INP, CLS en production sur les vrais utilisateurs → envoyer vers PostHog. Alertes si performance dégrade après un deploy.
  • 🔬
    Chrome DevTools — Performance Profiler Enregistrer un profil lors d'une interaction lente. Identifier les Long Tasks (>50ms). Cible : aucun Long Task sur le main thread.
  • 🗄️
    Supabase Logs + supabase inspect supabase inspect db --verbose → identifier les requêtes lentes. Objectif : aucune query > 50ms en P99.

// 08 — Claude Code Autonome

Comment configurer Claude Code pour qu'il maintienne la performance automatiquement

Hook PostToolUse — Vérification auto

À chaque modification de fichier, Claude Code vérifie automatiquement si un import lourd a été ajouté ou si une image n'utilise pas next/image.

# .claude/hooks/check-perf.js # Déclenché après chaque Write/Edit # Vérifier : imports lourds sans lazy loading if grep -q "import.*motion" $FILE; then if ! grep -q "dynamic.*motion" $FILE; then echo "⚠️ Motion devrait être un dynamic import" fi fi # Vérifier : <img> au lieu de next/image if grep -q "<img " $FILE; then echo "❌ Utiliser <Image> de next/image" fi

Slash command /dermotec-perf

Audit de performance complet avant chaque deploy. Déclenche 3 subagents en parallèle : bundle analysis, DB queries, image optimization.
